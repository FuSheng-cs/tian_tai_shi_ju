# 《天台十句》优化需求文档

> 基于 2026-04-30 对当前仓库的前端、后端、Prompt、文档和资源配置的代码审阅整理。本文档面向后续迭代排期，重点描述“需要优化什么、为什么优化、验收标准是什么”。

## 1. 文档目标

当前项目已经完成 AI 叙事游戏的核心闭环：标题页、主线十句话对话、动态好感度、提示、三结局、成就、后日谈聊天、前后端分离和本地存档。

下一阶段优化目标不是大改玩法，而是把现有闭环做稳：

- 让玩家首次进入就能顺利体验真实 AI 对话。
- 让核心规则、存档、结局、解锁关系可信。
- 让 Prompt、文档、资源、工程配置保持一致。
- 让项目从“高质量 demo”提升到“可公开展示和持续迭代的作品”。

## 2. 优先级定义

- **P0 必须修复**：影响主流程可玩性、核心规则可信度或线上展示效果。
- **P1 重要优化**：不一定阻断游玩，但会明显影响体验完整度、复玩价值或维护成本。
- **P2 体验增强**：提升沉浸感、可访问性、表达质量和长期扩展能力。
- **P3 后续探索**：需要更多设计或资源投入，不建议在基础稳定前优先做。

## 3. P0 必须修复需求

### \RQ-P0-01：保证公开环境具备真实 AI 试玩能力

**现状问题**

后端支持玩家自带 Key 和服务端兜底 Key，但如果线上环境没有配置 `LLM_API_KEY`，主线对话、提示和后日谈会退回模拟回复。对外展示时，玩家可能无法体验项目真正价值。

**涉及位置**

- `backend/config/config.go`
- `backend/handlers/game.go`
- `legacy_vue/src/views/SettingsView.vue`

**优化需求**

1. 为公开演示环境配置受控的服务端兜底模型。
2. 当前端没有填写 Key 时，默认走服务端兜底配置。
3. 设置页明确提示当前处于哪种模式：
   - 服务端试玩模式
   - 玩家自带 Key 模式
   - 模拟回复模式
4. 若无真实模型可用，首页或设置页应明确提示“当前为模拟体验”。

**验收标准**

- 不填写 API Key 时，公开环境至少能完成一局真实 AI 主线对话。
- `/api/chat`、`/api/hint`、`/api/chat-after` 都能在服务端配置存在时调用真实模型。
- 模型不可用时，玩家能看到明确提示，而不是误以为完整功能已失效。

---

### \RQ-P0-02：修复存档核心状态不完整问题

**现状问题**

存档保存了 `roundCount`、`hintCount`、`messages`、`isEnding`、`endingType`，但没有保存 `affection`。好感度是核心机制，读档后丢失会导致规则状态失真。

**涉及位置**

- `legacy_vue/src/modules/SaveSystem.ts`
- `legacy_vue/src/store/gameStore.ts`
- `legacy_vue/tests/SaveSystem.test.ts`

**优化需求**

1. 将 `affection` 纳入存档状态。
2. 建立明确的 `PersistedGameState` 类型，避免用 `any` 传递存档状态。
3. 读档时恢复完整核心状态。
4. 增加测试覆盖：保存后读回的核心状态必须一致。

**验收标准**

- 设置 `affection > 0` 后保存、重置、读档，`affection` 保持不变。
- 测试覆盖 `roundCount`、`hintCount`、`affection`、`messages`、`isEnding`、`endingType`。
- `loadState()` 不再依赖裸 `any`。

---

### \RQ-P0-03：避免等待 LLM 回复时保存半完成状态

**现状问题**

玩家发送消息后，系统会先扣回合并追加用户消息，再等待 LLM 回复。此时保存会得到“用户发言已保存、AI 回复未保存”的半完成档。

**涉及位置**

- `legacy_vue/src/views/GameView.vue`
- `legacy_vue/src/store/gameStore.ts`
- `legacy_vue/src/modules/SaveSystem.ts`

**优化需求**

1. `gameStore.isWaiting` 为 true 时禁用保存按钮。
2. `saveGame()` 内部也要做保护，避免未来其他入口绕过按钮状态。
3. 提示玩家“请等待当前回复完成后再保存”。

**验收标准**

- LLM 回复等待期间无法保存。
- 保存后读档不会出现只有玩家消息、没有对应 AI 回复的异常回合。

---

### RQ-P0-04：为后日谈增加解锁守卫

**现状问题**

`/chat-after` 路由可直接访问，未校验是否达成“相识”结局。后日谈作为成功结局奖励，当前可被绕过。

**涉及位置**

- `legacy_vue/src/router/index.ts`
- `legacy_vue/src/modules/AchievementTracker.ts`
- `legacy_vue/src/views/ChatAfterStoryView.vue`

**优化需求**

1. 进入 `/chat-after` 前校验是否已解锁 `end_acquaintance`。
2. 未解锁时重定向到首页或成就页。
3. 给出明确提示：需要达成“相识”结局后解锁。

**验收标准**

- 未达成“相识”结局时无法直接打开后日谈页。
- 达成“相识”结局后可以正常进入。
- 刷新页面后仍能基于本地成就状态判断访问权限。

## 4. P1 重要优化需求

### \RQ-P1-01：开放多栏位存档 UI

**现状问题**

底层 `SaveSystem.getSlots()` 支持 3 个栏位，但游戏页保存和标题页读取都固定使用栏位 1。

**涉及位置**

- `legacy_vue/src/modules/SaveSystem.ts`
- `legacy_vue/src/views/GameView.vue`
- `legacy_vue/src/views/StartView.vue`

**优化需求**

1. 做统一的存档/读档弹窗。
2. 显示 3 个手动存档栏位。
3. 每个栏位展示：保存时间、剩余回合、当前对话数、最近一句摘要、是否已结局。
4. 覆盖已有存档前二次确认。
5. 可选增加删除存档功能。

**验收标准**

- 玩家可以选择保存到 1、2、3 任意栏位。
- 玩家可以从 1、2、3 任意有效栏位读档。
- 坏档或空档不会导致页面崩溃。

---

### RQ-P1-02：存档格式版本化与结构校验

**现状问题**

当前 payload 没有版本号，读档时只校验 CRC32，不校验状态结构。未来状态升级后旧档兼容风险较高。

**涉及位置**

- `legacy_vue/src/modules/SaveSystem.ts`

**优化需求**

1. payload 增加 `version`、`savedAt`。
2. 增加存档结构校验函数。
3. `getSlots()` 对单个坏档容错，不能因为一个坏档影响整个列表。
4. 明确 CRC32 只作为完整性校验，不作为安全防作弊机制。

**验收标准**

- 非法 JSON、错误字段、错误消息结构都无法读入游戏状态。
- 单个坏档不影响其他栏位展示和读取。
- 未来可根据 `version` 增加迁移逻辑。

---

### RQ-P1-03：统一结局 ID、资源和成就定义

**现状问题**

`GameView.vue` 中仍有历史结局分支，如 `end_true_release`、`end_self_deception`、`end_eternal_cage`，但当前真实结局只有 `end_death`、`end_disappear`、`end_acquaintance`。同时“相识”结局没有明确在当前分支里使用专属成功 CG。

**涉及位置**

- `legacy_vue/src/views/GameView.vue`
- `legacy_vue/src/modules/AchievementTracker.ts`
- `legacy_vue/public/assets/images/`
- `docs/product/storyline_and_lore.md`

**优化需求**

1. 清理不再使用的历史结局分支。
2. 建立单一结局定义表，统一 ID、中文名、成就、CG、后续入口。
3. 为 `end_acquaintance` 明确配置成功结局 CG。
4. 文档、Prompt、代码中的结局名称保持一致。

**验收标准**

- 代码中不存在不可达的旧结局分支。
- 三个结局都能对应明确的视觉资源和成就。
- 文档和代码的结局 ID 一致。

---

### RQ-P1-04：完善设置持久化与设置生效闭环

**现状问题**

LLM 配置已经独立持久化，但字体、行高、音量、屏幕阅读器模式等设置没有完整持久化闭环。刷新后可能回到默认值。

**涉及位置**

- `legacy_vue/src/store/settingsStore.ts`
- `legacy_vue/src/views/SettingsView.vue`
- `legacy_vue/src/App.vue`
- `legacy_vue/src/modules/AudioManager.ts`

**优化需求**

1. 设置项保存到 `localStorage`。
2. 应用启动时恢复设置。
3. 音量调整应实时影响当前 BGM 和音效。
4. 字体、行高应对主线对话、提示、后日谈等主要文本区域生效。
5. 暂未实现的设置项不要展示，或补齐实现。

**验收标准**

- 刷新页面后设置保持不变。
- 调整音量后当前播放的 BGM 立即变化。
- 调整字体和行高后主要文本区域可见变化。

---

### RQ-P1-05：修复资源引用缺漏

**现状问题**

部分代码和配置引用的资源在 `public/` 中不存在，例如：

- `AudioManager` 引用 `/assets/audio/ui_click.wav`，当前资源目录未见该文件。
- PWA manifest 引用 `/pwa-192x192.png`、`/pwa-512x512.png`，当前 `public/` 未见对应文件。

**涉及位置**

- `legacy_vue/src/modules/AudioManager.ts`
- `legacy_vue/vite.config.ts`
- `legacy_vue/public/`

**优化需求**

1. 补齐缺失资源，或把引用改为真实存在的资源。
2. 统一 favicon、PWA icon、音效资源命名。
3. 增加资源引用检查清单或脚本。

**验收标准**

- 浏览器控制台不再出现资源 404。
- 点击音效、打字音效、BGM 均可按设置播放。
- PWA manifest 中声明的图标真实存在。

---

### RQ-P1-06：修复或收敛 Lint 配置

**现状问题**

`package.json` 使用 `eslint .`，依赖为 ESLint 10，但项目仍存在旧式 `.eslintrc` 配置风险。静态检查如果不可用，会影响后续多人协作和质量控制。

**涉及位置**

- `legacy_vue/package.json`
- `legacy_vue/.eslintrc.cjs`

**优化需求**

1. 迁移到 ESLint flat config，或锁定到兼容 `.eslintrc` 的 ESLint 版本。
2. 明确 lint 规则，尤其是 TypeScript `any` 的处理策略。
3. 将 lint 纳入提交前或构建前检查。

**验收标准**

- `npm run lint` 可稳定运行。
- 新增代码不会引入明显类型和 Vue 模板错误。

## 5. P2 体验增强需求

### \RQ-P2-01：用游戏内轻提示替代 `alert`

**现状问题**

保存成功、保存失败、读档失败等反馈使用浏览器 `alert`，会打断沉浸感。

**优化需求**

1. 增加全局 toast 或局部提示组件。
2. 保存、读档、连接测试、后日谈解锁失败等提示统一使用游戏内反馈。
3. 文案可结合游戏氛围，但必须保持信息明确。

**验收标准**

- 主流程不再使用阻塞式 `alert`。
- 玩家能明确知道操作结果。

---

### RQ-P2-02：增加“继续游戏”入口

**现状问题**

标题页只有“开始游戏”和“读取存档”。如果玩家有最近存档，缺少更直接的继续入口。

**优化需求**

1. 标题页检测最近有效存档。
2. 存在有效存档时显示“继续游戏”。
3. “继续游戏”读取最近一次保存的栏位。
4. “读取存档”仍进入完整栏位列表。

**验收标准**

- 有存档时可一键继续。
- 没有存档时不展示或置灰“继续游戏”。

---

### RQ-P2-03：增强提示系统的分层反馈

**现状问题**

当前提示系统能给方向，但还没有把玩家问题分层，例如“太说教”“太急”“没有回应她刚刚的话”“缺少共情”等。

**涉及位置**

- `backend/llm/service.go`
- `legacy_vue/src/store/gameStore.ts`
- `legacy_vue/src/views/GameView.vue`

**优化需求**

1. 提示 Prompt 增加不同提示类型。
2. 提示结果保持短句，不直接给标准答案。
3. 可在 UI 上区分提示风格，例如“倾听方向”“风险提醒”“关系观察”。

**验收标准**

- 提示能根据对话上下文指出不同问题。
- 提示不泄露可直接复制的答案。
- 提示仍然保持 20 字左右的轻量表达。

---

### RQ-P2-04：强化后日谈与主线结局的承接

**现状问题**

后日谈固定以“我到家了”“今天天台的风还挺大的”开场，没有承接玩家刚刚达成“相识”的具体过程。

**涉及位置**

- `legacy_vue/src/views/ChatAfterStoryView.vue`
- `legacy_vue/src/modules/LLMService.ts`
- `backend/handlers/game.go`
- `backend/llm/service.go`

**优化需求**

1. 将主线最后几轮对话摘要或关键状态带入后日谈。
2. 后日谈开场根据玩家表现轻微变化。
3. 后日谈 Prompt 明确关系边界：不是恋爱模拟器，而是“被救下后仍在慢慢恢复的人”。

**验收标准**

- 玩家能感受到后日谈来自刚刚那一局，而不是独立聊天页。
- 后日谈不会与主线人设割裂。

---

### RQ-P2-05：接入屏幕阅读器友好模式

**现状问题**

设置中存在“屏幕阅读器友好模式”，但当前没有形成完整功能闭环。

**涉及位置**

- `legacy_vue/src/store/settingsStore.ts`
- `legacy_vue/src/views/SettingsView.vue`
- `legacy_vue/src/components/TypewriterText.vue`
- 各主要视图模板

**优化需求**

1. 开启后关闭或加速打字机效果。
2. 为关键按钮和动态文本增加合适的 aria 属性。
3. 避免只通过颜色、背景图传达关键信息。
4. 后日谈消息区域增加合理的 live region 策略。

**验收标准**

- 开启模式后文本可立即完整呈现或显著减少等待。
- 键盘与读屏用户可以理解主要流程。

## 6. P3 后续探索需求

### RQ-P3-01：主线和后日谈上下文管理

**现状问题**

当前对话历史基本原样传给模型。主线回合短，问题暂时可控；后日谈是开放聊天，长期会导致 token 成本、延迟和上下文漂移上升。

**优化需求**

1. 主线保留完整短历史，必要时增加关键事件摘要。
2. 后日谈采用“最近消息 + 关系记忆摘要”的结构。
3. 设定最大上下文长度，超出后摘要压缩。

**验收标准**

- 后日谈长聊后响应成本和延迟可控。
- 角色仍能记住关键关系信息。

---

### RQ-P3-02：流式输出与打字机融合

**现状问题**

当前前端等待完整回复后再打字机显示，没有真正利用 LLM 流式输出。

**优化需求**

1. 后端支持 OpenAI-compatible streaming。
2. 前端逐步接收并显示角色回复。
3. 保留控制标签解析能力，避免 `[好感度+5]`、`[结局:*]` 被直接展示给玩家。

**验收标准**

- 玩家能看到角色逐步开口。
- 好感度和结局标签仍能被正确解析并隐藏。
- 网络中断时有合理恢复或错误提示。

---

### RQ-P3-03：结局回放与局后总结

**现状问题**

成就系统已有雏形，但结局后的回看价值较弱。

**优化需求**

1. **结局页展示本局摘要：使用了几句话、触发几次好感、关键转折句。**
2. **给出简短局后评语。**
3. **成就页可查看已达成结局的回忆片段。**

**验收标准**

- 玩家达成结局后有明确复盘内容。
- 多结局收集动力增强。

## 7. 非功能性需求

### 7.1 稳定性

- API 请求失败时，前端必须恢复可操作状态。
- 存档损坏不能导致页面崩溃。
- 模型返回异常格式时，游戏应有兜底处理。

### 7.2 安全与隐私

- 玩家 API Key 只保存在本地浏览器和当前请求链路中，不写入日志。
- 后端日志不得输出完整 API Key。
- 自定义 Base URL 作为高级功能，需要明确提示玩家风险。
- 公开环境服务端 Key 应配合额度限制、频率限制或访问控制。

### 7.3 可维护性

- Prompt、文档和代码中的角色设定、结局名称、机制标签必须同步。
- 核心状态类型应集中定义，减少重复手写字段。
- 测试至少覆盖 gameStore、SaveSystem、LLMService 请求组装和关键路由守卫。

### 7.4 可访问性

- 所有核心按钮可键盘操作。
- 重要动态反馈不能只依赖颜色。
- 打字机效果应可跳过，屏幕阅读器模式下应弱化。

## 8. 建议迭代顺序

### 第一阶段：稳定主流程

1. 配置真实 AI 试玩能力。
2. 修复存档 `affection` 缺失。
3. 禁止等待中保存。
4. 增加后日谈路由守卫。

### 第二阶段：补齐系统闭环

1. 多栏位存档 UI。
2. 存档版本化和结构校验。
3. 设置持久化。
4. 结局 ID、资源、成就统一。
5. 修复资源缺漏。

### 第三阶段：提升展示质量

1. Toast 替代 alert。
2. 继续游戏入口。
3. 提示系统分层。
4. 后日谈承接主线。
5. 屏幕阅读器友好模式闭环。

### 第四阶段：探索 AI 原生增强

1. 后日谈长期记忆摘要。
2. 流式输出。
3. 结局回放和局后总结。

## 9. 总结

《天台十句》的核心优势已经明确：它把 LLM 的自由输入、延迟、成本和模糊判断都转化成了玩法的一部分。当前最值得投入的方向不是增加复杂系统，而是增强“可信度”和“完成度”：

- 规则可信：好感、回合、结局、存档必须一致。
- 体验可信：等待、提示、后日谈都要服务叙事，而不是暴露工程痕迹。
- 工程可信：配置、资源、测试、Lint 和文档要能支撑持续迭代。
